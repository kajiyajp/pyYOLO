## セッション開始時（最初に必ず読む）

再開・引継ぎ時は、以下の順で読んでから作業に入る：

1. **README.md** … リポ全体の目的・構成・実験結果
2. **RESUME.md** … 前回やったこと・次にやること・懸念点
3. **ROADMAP.md** … どのPhaseまで完了し、次に何をするか
4. 必要に応じてコードベースを確認

この3ファイルで前回からの引継ぎが完結するように、各ドキュメントを常に最新に保つこと。

---

## コミュニケーション

- 回答は常に**日本語**で行う
- 専門用語は初出時に一言補足をつける
- 長い説明より短いコードと実行例を優先する
- 実行前に「何をするか」を箇条書きで提示してから着手する
- 詰まったときは原因と解決策を2〜3案提示する（1つに絞らない）

## ブランチ戦略

- `main` は常に動く状態を保つ
- 作業は必ず **feature ブランチ** で行う
- ブランチ命名規則：`feature/機能名` （例：`feature/annotation-tool`）
- PRやissueを使わない場合でも、マージ前に動作確認を行う
- `main` への直接コミットは禁止（緊急修正を除く）

```bash
# 作業開始時の手順
git checkout -b feature/xxx
# 作業・コミット
git checkout main
git merge feature/xxx
git push origin main
```

## コーディング規約

- コメントは**日本語**
- 関数・メソッドには役割を**1行コメント**で簡潔に記述する
- ファイルヘッダーに作成日と目的を記述する（既存フォーマットを踏襲）
- `TODO` / `FIXME` はコメントで残す（無断で消さない）
- 不要なコメントアウトコードは残さない

```python
# 良い例
def preprocess(frame):
    """画像をYOLO入力形式に変換する"""
    ...

# 悪い例（何をするかしか書いていない）
def preprocess(frame):
    # フレームをリサイズしてnormalizeする
    ...
```

## ドキュメント管理

### ROADMAP.md
- `origin main` へ push する**前**に必ず更新する
- 完了したフェーズは `[x]` にチェックを入れる
- 新しいフェーズ・タスクが生まれたら追記する

### RESUME.md
- 作業を区切るとき（セッション終了・テーマ切替）に更新する
- 記載内容：
  - 最後にやったこと
  - 次にやること（具体的に）
  - 未解決の問題・懸念点
- 再開時はROADMAP.mdとRESUME.mdとコードベースを見て引き継ぐ

### TODO.md
- 残タスクを優先度別に管理する
- 完了したらその場で `[x]` にする

## プロジェクト固有の知識

### アーキテクチャ方針
- **YOLO（onnxruntime）**：ターゲット発見・BB出力・ROI自動設定
- **既存CV（PointLock側）**：精密座標算出（±0.1px）
- 役割分担を崩さない。YOLOに精密座標を求めない

### ライセンス方針
- `ultralytics`（AGPL-3.0）は**社内学習専用**。製品には同梱しない
- 製品推論は `onnxruntime`（MIT）＋ `best.onnx` のみ
- 製品化時はUltralytics Enterpriseライセンス交渉を行う

### 対象ターゲットクラス
| クラスID | クラス名 | 素材・特徴 |
|---------|---------|-----------|
| 0 | `l_mark` | 白L字マーク（コンクリート等） |
| 1 | `cross` | ケガキ十字（zinc処理SM490鋼板） |
| 2 | `circle_cross` | 白丸＋ケガキ十字（zinc処理SM490鋼板） |
| 3 | `l_mark_black` | L字マーク（黒皮SS400鋼板） |

### モデル比較結果（2026-06-28時点）
| モデル | 推論時間avg | mAP50 | 後処理 |
|--------|-----------|-------|--------|
| YOLO11n-ONNX | 22.6ms | 0.995 | NMSあり |
| YOLO26n-ONNX | 19.5ms | 0.995 | NMSフリー |
