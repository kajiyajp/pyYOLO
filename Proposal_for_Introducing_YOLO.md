# YOLO導入提案 — PointLock ROADMAPフェーズ見直し

> 作成：2026-06-28  
> 作成者：kajiyajp（pyYOLO実験セッションより）  
> 対象：app-PointLock ROADMAP.md フェーズ3

---

## 1. 背景

PointLock ROADMAP フェーズ3 には以下の記述がある：

> 蓄積データで**軽量検出モデルを学習**（目標の自動発見・分類）→ ROIクリック削減  
> ※ 中心の最終精度は従来CVが担保。MLは「探す・分類する」役割

この「軽量検出モデル」が **YOLO（You Only Look Once）** であることが  
pyYOLO リポでの実験（2026-06-28）で確認できた。  
本提案はその実験結果をもとに、YOLO導入をフェーズ3から**フェーズ1.5へ前倒し**することを提案する。

---

## 2. 現状の課題

| 課題 | 内容 |
|------|------|
| ROI手動指定 | 作業者が毎回ドラッグで指定する必要がある |
| ターゲット種別手動選択 | L字・十字・白丸+十字を都度コンボで選ぶ必要がある |
| 多様なターゲット | zinc処理SM490のケガキ十字・白丸+十字、黒皮SS400のL字など種別が多い |
| 検出安定性 | look-then-moveで根治したが、ROI設定の精度はまだ人間依存 |

---

## 3. YOLOで解決できること

```
カメラ映像
    ↓
YOLO（軽量検出モデル）
    ├─ 「L字マーク発見 → ROI自動設定 + method=white_corner」
    ├─ 「ケガキ十字発見 → ROI自動設定 + method=cross」
    └─ 「白丸+十字発見 → ROI自動設定 + method=white_cross」
    ↓
既存CV（detect_white_corner 等）
    └─ ROI内で精密座標を算出（角Rを含む2辺延長交点）
    ↓
look-then-move で収束
```

### 作業者の操作変化

| 操作 | 現在 | YOLO導入後 |
|------|------|-----------|
| ROIドラッグ指定 | 毎回必要 | **不要** |
| ターゲット種別選択 | 毎回必要 | **不要** |
| 引継ぎ開始ボタン | 必要 | 必要（安全のため残す） |

---

## 4. 役割分担（確定方針）

| 層 | ツール | 役割 |
|----|--------|------|
| 発見・分類 | **YOLO** | 画面内のターゲット種別を認識しROIを自動設定 |
| 精密検出 | **既存CV** | ROI内で角Rを考慮した交点座標を算出（±0.1px） |
| 収束制御 | **look-then-move** | 静止1ショット → エンコーダ閉ループ移動 |

> ※ YOLOは「探す」、CVは「正確に測る」の分業。精度はCVが担保するためYOLOに高精度は不要。

## 5. 実装方針

### PointLock内への内包

```
pypoint/
├── vision/
│   └── yolo_detector.py     # YOLO推論クラス（新規）
└── assets/
    └── models/
        └── best.pt           # 学習済みモデルを同梱
```

`requirements.txt` に `ultralytics>=8.4.0` を追加するだけで動作する。

### クラス定義

```yaml
nc: 4
names:
  0: l_mark         # L字マーク（白・黒皮SS400）
  1: cross          # ケガキ十字（zinc処理SM490）
  2: circle_cross   # 白丸＋ケガキ十字（zinc処理SM490）
  3: l_mark_black   # 黒皮SS400 L字（必要に応じて追加）
```

### PointLockへの接続イメージ

```python
# vision/yolo_detector.py
results = model(frame)
for box in results[0].boxes:
    class_id = int(box.cls[0])
    method = CLASS_TO_METHOD[class_id]   # 自動でmethod選択
    roi = box_to_roi(box)
    session.set_roi(roi, deliberate=True)
    session.set_method(method)
```

## 6. 教師データ収集戦略（前倒しの核心）

### 問題
現在は教師データがない。フェーズ3まで待つと実機稼働後になる。

### 解決策：学習モードを先に作る

```
① 実機稼働前に各ターゲットを動画撮影（5〜10分/種別）
      L字マーク・ケガキ十字・白丸+十字・黒皮L字

② PointLockに簡易アノテーションUIを追加（学習モード）
      datalogのPNG + PySide6で矩形描画 → YOLO形式.txt保存

③ 10〜20枚アノテーション → 粗いモデルを学習

④ 実機稼働後にdatalogへPNGが自動蓄積

⑤ 蓄積トリガーで自動再学習 → best.pt自動更新

⑥ 使えば使うほど精度が上がる継続学習ループ
```

### アノテーションUIの実装コスト
PySide6の既存コード（`calib_wizard.py`）が類似構造を持つため、  
Claude Codeで比較的短期間に実装可能。

## 7. モデル管理・更新運用

学習と推論を以下のように分離する。

| 作業 | 場所 | 担当 |
|------|------|------|
| データ収集・アノテーション・学習 | 開発者の調査・実験環境 | 開発者 |
| 推論（best.pt を内包して実行） | app-PointLock | アプリ本体 |

### モデル更新フロー

```
新ターゲット追加 or 精度改善が必要
↓
学習環境でアノテーション → 追加学習 → best.pt 生成
↓
app-PointLock の assets/models/best.pt を差し替え
↓
コード変更不要・次回起動から自動反映
```

> ポイント：モデルファイル（best.pt）を差し替えるだけで更新完了。  
> アプリ本体の改修・再ビルドは不要。

## 9. 今日の実験結果（pyYOLO）

| 項目 | 結果 |
|------|------|
| 使用モデル | YOLOv11n（260万パラメータ・GPU不要） |
| 教師データ | L字マーク動画から100枚抽出・手動アノテーション |
| アノテーション確認 | 矩形がL字を正確に囲んでいることを可視化で確認 |
| 学習時間 | CPU（Core i7-11700K）で約6分 |
| mAP50 | 0.995（val） |
| 推論速度 | 42ms/枚 → リアルタイム十分 |
| 課題 | 同一動画100枚では均質すぎ・多様な教師データが必要 |

**結論：YOLOはPointLockの検出層に技術的に統合可能であり、実用レベルで動作することを確認した。**
