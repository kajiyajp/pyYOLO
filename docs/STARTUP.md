## 0. ひとことで言うと

「PointLock（産業用CNCビジュアルサーボ）にYOLOを導入できるか」を、
個人リポ pyYOLO で**ゼロから検証**し、
**撮影→注釈→学習→ONNX変換→推論まで1ツールで完結する社内環境**を作った。

---

## 1. 出発点

- きっかけ：Python + YOLO を学びたい。PointLockへの応用を探りたい。
- PointLockとは：カメラ画像でCNC軸をターゲットへ自動収束させるビジュアルサーボ。
  - 動作：look-then-move（静止フレームで検出→ΔX/ΔY算出→エンコーダ閉ループ移動）
  - 既存CV：`detect_white_corner`（L字の直線を延長し交点を精密座標化、角Rに対応）
- 最初の素材：PointLockのサンプル動画1本（816x684, 523フレーム）

---

## 2. 役割分担の発見（設計の核）

```
ライブ映像
   ↓
YOLO   … ターゲットを「見つける」→ BB/クラス → ROI自動設定
   ↓
既存CV … その範囲を「測る」→ ±0.1px精密座標
   ↓
ΔX/ΔY → CNC軸フィードバック
```

**「YOLOは見つける、CVは測る」** — この分担を崩さないことが全体の指針になった。

---

## 3. 最初の実験と失敗

- 動画から100フレーム抽出 → Make Senseでl_mark注釈 → YOLO11n学習
- **失敗**：推論で検出ほぼ0。原因は「同じ動画100枚＝均質すぎ」。
- 学び：**データの量と多様性が命**。

---

## 4. モデルの比較検証

| 検証 | 結果 |
|------|------|
| YOLO11n 学習 | mAP50 0.995 |
| Tracking | 全523フレームでID:1一貫 |
| OBB（回転BB） | mAP50 0.995・角度検出 |
| ONNX推論（ultralytics不使用） | 22.6ms/frame |
| YOLO26n（NMSフリー） | 19.5ms/frame・最速15.7ms |

- **YOLO26**は実在（YOLO11の後継・NMSフリー・エッジ最適化）。
- 速度・精度ともYOLO11nと同等〜やや上。

## 5. ライセンスの整理（重要）

- ultralyticsは **AGPL-3.0**。製品に同梱するとソース公開義務が発生。
- PointLockは顧客納入品（クローズドソース）なので**同梱したくない**。
- **解決策**：
  ```
  社内：ultralytics(AGPL)で学習 → best.onnx を作る（配布しない）
  製品：onnxruntime(MIT)で best.onnx を推論（ultralytics非同梱）
  ```
- `onnx_detector.py` は onnxruntime のみで動く＝PointLock移植の土台。
- 製品化時はUltralytics Enterpriseライセンスをアジア担当と交渉予定。
  - 15台買い切りの予想：一括150〜400万円（高めという感触）。
  - ONNXの重みだけなら本当に必要かも要確認。

## 6. 作ったもの：社内アノテーションツール（annotation_tool.py）

PySide6 + OpenCV + onnxruntime。**4タブで撮影〜推論まで完結**。

| タブ | 機能 |
|------|------|
| カメラ撮影 | UVC（Camo経由iPhoneで動作確認）・緑グロー枠・Space連続撮影 |
| アノテーション | 矩形描画・クラストグル(F1〜)・単一/複数表示(W)・注釈枠/未済表示 |
| 学習・変換 | データセット作成→学習→ONNX変換（Python有無で自動判定） |
| 推論 | .onnx読込→ライブ/静止画でリアルタイム検出 |

- ショートカット：Space撮影 / Ctrl+C停止 / S保存 / A,D画像 / W表示 / F1〜クラス
- クラスは `classes.json` で設定可能（設定画面・再起動後反映）
- PyInstallerでexe化（社内配布用）。学習はPython(ultralytics)が要るため
  配布先で自動的に有効/無効を判定。

## 7. データ量の実証（この日の最大の学び）

実機（手持ちで撮影）の画像で撮影→注釈→学習→推論ループを検証：

| 指標 | 22枚 | 80枚 |
|------|------|------|
| mAP50 | 0.82 | 0.92 |
| conf>=0.25で検出 | 17/22 | 80/80 |
| 平均最大信頼度 | 0.06〜0.18 | **0.70** |

→ **枚数とバラつきを増やすだけで信頼度が0.1→0.70に激変。**
   実運用は各クラス80枚以上・角度/距離/明るさを変えるのが目安。

## 8. フォルダ運用の方針

- captures は**モデル（用途）ごとにフォルダ分け**（marks / holes …）。
- 穴（φ10〜φ30）は **1クラス "hole"** でYOLOが検出、径はCVが測る。
  サイズ違いをクラス分けしない（「見つける/測る」分担のまま）。
- 将来は「用途別に管理 → 学習時に1つの多クラスモデルに統合」が運用しやすい。

---

## 9. 次の最優先：Keypoint検証

- 目的：YOLOでL字頂点を直接検出し、精度/確度を測る。
- **本命の価値（現場の実情報で判明）**：
  ```
  既存CVは精度は出るが、ライブ映像で誤認識が激しく不安定・調整に時間がかかる。
  → YOLO導入の狙いは「サブpx精度」より【ライブ映像での安定性・誤検出耐性】。
  → YOLOが安定して捕捉(ゲート)→CVが安定ROI内だけ追い込む → 暴れと調整時間を抑える。
  ```
- 検証の最重要指標：**ライブ安定性**（連続フレームで検出が途切れない率）・誤検出率・ジッター・pixel誤差。
- 準備：Keypointアノテーション（頂点1点指定）＋誤差測定スクリプト。
- 手応え：Trackingで全523フレームID一貫・ライブ推論20ms は確認済み。
